# Codebase Concerns

**Analysis Date:** 2026-06-07

## Tech Debt

**Database connection uses non-null assertion:**
- Issue: `process.env.DATABASE_URL!` with lazy Proxy — error only surfaces on first DB query, not at startup
- Files: `packages/db/src/index.ts`
- Why: Intentional lazy init to prevent crash on import when DATABASE_URL not set (valid pattern)
- Impact: Confusing error traces for new users — no helpful message at startup
- Fix approach: Add startup validation with a clear error message, or wrap in try/catch with user-facing guidance

**API server has no env validation:**
- Issue: `apps/api/src/index.ts` spins up without checking required env vars
- Files: `apps/api/src/index.ts`
- Why: Template-level — validation left to user
- Impact: Runtime errors deep in DB calls rather than startup failure
- Fix approach: Add `import { env } from '@repo/env'` at top of `apps/api/src/index.ts`

**Schema duplication risk (users table):**
- Issue: Two `users` table definitions — `packages/db/src/schema/index.ts` (example) and `packages/auth/src/schema.ts` (auth schema)
- Files: `packages/db/src/schema/index.ts`, `packages/auth/src/schema.ts`
- Why: Default schema is a placeholder; auth schema is opt-in
- Impact: Drizzle migration conflict if user enables auth without removing the example users table
- Fix approach: `docs/AUTH.md` warns about this. Consider removing example `users` table or adding a strong inline comment

## Known Issues / Gotchas

**PWA manifest name is hardcoded:**
- Issue: `apps/web/vite.config.ts` hardcodes `name: 'shadcn-preset'` in PWA manifest
- Files: `apps/web/vite.config.ts`
- Impact: After running `setup.js` to rename project, installed PWA still shows "shadcn-preset"
- Workaround: Manually update PWA `name` and `short_name` in `vite.config.ts`

**`routeTree.gen.ts` stale without dev server:**
- Issue: File is auto-generated by TanStack Router Vite plugin — only updates when dev server is running
- Files: `apps/web/src/routeTree.gen.ts`
- Impact: New routes don't appear until dev server runs at least once; TypeScript errors if routes are added offline
- Workaround: Always run `pnpm dev` before working on routes

**Dev port 3001 is hardcoded:**
- Issue: `serve({ fetch: app.fetch, port: 3001 })` in dev.ts
- Files: `apps/api/src/dev.ts`
- Impact: If port 3001 is in use, server fails without fallback or configurable override
- Fix approach: `const port = parseInt(process.env.PORT ?? '3001'); serve({ port })`

**`cn` import path is non-obvious:**
- Issue: Must import from `@repo/ui/src/lib/utils` not `@repo/ui/lib/utils`
- Files: `packages/ui/tsconfig.json` (alias resolves to root, not src/)
- Impact: Type errors for new users who follow generic shadcn import patterns
- Documented in: `CLAUDE.md` Gotchas section

## Missing Coverage

**No tests at all:**
- What's missing: Zero test files across the entire monorepo
- Files: All `*.ts`/`*.tsx` source files
- Risk: Regressions undetectable — changes to DB proxy, auth, or routing break silently
- Priority: Medium (template — user adds their own tests)
- See: `TESTING.md` for recommended setup

**No error boundaries:**
- What's missing: No React Error Boundary or TanStack Router `errorComponent` in `apps/web/src/routes/__root.tsx`
- Files: `apps/web/src/routes/__root.tsx`
- Risk: Unhandled component errors show blank screen with no user feedback
- Fix approach: Add TanStack Router `ErrorComponent` to `createRootRoute`

**No QueryClient error handler:**
- What's missing: No global `onError` in `apps/web/src/lib/query.ts`
- Files: `apps/web/src/lib/query.ts`
- Risk: Failed queries silently ignored unless each `useQuery` caller handles errors
- Fix approach: Add `onError` callback that triggers Sonner toast (Sonner already installed)

**No API error middleware:**
- What's missing: No global error handler in Hono app
- Files: `apps/api/src/index.ts`
- Risk: Uncaught route handler errors return 500 with no structured response
- Fix approach: Add `app.onError((err, c) => c.json({ error: err.message }, 500))`

## Fragile Areas

**`apps/web/src/routeTree.gen.ts`:**
- Why fragile: Auto-generated, `@ts-nocheck` at top, easy to accidentally edit
- Common failures: Manual edits overwritten by dev server; stale file causes TypeScript errors
- Safe modification: Never edit directly — add/modify files in `apps/web/src/routes/`

**`packages/ui/src/index.ts` barrel export:**
- Why fragile: 200+ named exports — easy to miss exporting new components after install
- Common failures: Component installed in `packages/ui/src/components/ui/` but not exported from index
- Safe modification: Always add export after `npx shadcn add [component]`

**`packages/auth/` integration:**
- Why fragile: Not tested with the rest of the boilerplate; opt-in with manual steps
- Common failures: Schema conflicts (see Tech Debt above), missing env vars
- Safe modification: Follow `docs/AUTH.md` exactly; resolve schema conflicts first

## Scaling Limits

**Vercel Hobby plan constraints:**
- Current capacity: 10s function timeout, 100GB-hrs/mo compute
- Limit: Edge Functions may timeout for heavy DB operations
- Scaling path: Upgrade to Vercel Pro ($20/mo)

**Neon free tier:**
- Current capacity: 512MB storage, compute suspends after inactivity (cold start ~500ms)
- Limit: Not suitable for high-traffic production
- Scaling path: Neon Pro or migrate to dedicated PostgreSQL

## Dependencies at Risk

**better-auth (latest tag):**
- Risk: Pinned to `latest` — major version bumps could introduce breaking changes
- Impact: Auth stops working on `pnpm install` after breaking release
- Fix approach: Pin to specific version once integrated

**drizzle-orm and drizzle-kit (latest tag):**
- Risk: Same issue — `latest` tag in `packages/db/package.json`
- Impact: Schema generation or migration CLI changes break silently
- Fix approach: Pin to specific versions in production use
